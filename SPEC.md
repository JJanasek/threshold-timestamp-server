# FROST Threshold Timestamp Authority over Nostr — Rust Implementation

## Project overview

Build a distributed trusted timestamping service where the signing key is
never held by any single party. The private key is split into n Shamir shares
using the FROST threshold Schnorr signature scheme. Each share is held by an
independent signer node. Signing requires k-of-n nodes to cooperate over the
Nostr protocol — no direct connections between nodes, no static IPs required.
Mobile devices, home servers, and cloud VMs can all participate equally.

The final output is a signed TimestampToken: a JSON document containing the
document hash, UTC timestamp, signing policy, and a standard secp256k1 Schnorr
signature verifiable with just the group public key.

## FROST integration (frost-core crate)

Use the `frost-secp256k1` crate directly. Do not reimplement the crypto.
Wrap it in thin types that serialise cleanly to JSON for Nostr event content.

### Key generation

Use `frost_secp256k1::keys::generate_with_dealer` for the initial trusted-dealer
keygen. Produce a `KeyPackage` per signer and a `PublicKeyPackage` for the
coordinator and verifiers.

For each signer write a config file containing:

- Their `KeyPackage` serialised as JSON (keep secret, file mode 0600)
- Their Nostr nsec (generated fresh — this is their transport identity)
- The `PublicKeyPackage` (public, same for all)
- The coordinator's npub

For the coordinator write a config containing:

- The `PublicKeyPackage`
- A list of authorised signer npubs with their FROST signer identifiers
- The coordinator's own nsec
- Relay URLs

### Signing rounds

Round 1: `frost_secp256k1::round1::commit(key_package.signing_share())`
Returns `(SigningNonces, SigningCommitments)`. Store nonces in memory keyed by
session_id. Publish `SigningCommitments` as Nostr kind 20002.

Round 2: `frost_secp256k1::round2::sign(signing_package, nonces, key_package)`
`signing_package` is built by the coordinator from the message and all
collected `SigningCommitments`. Returns `SignatureShare`. Publish as kind 20004.

Aggregate: `frost_secp256k1::aggregate(signing_package, &shares, &public_key_package)`
Returns a `Signature`. Verify with `frost_secp256k1::verify_group_signature`.

## Nostr event kinds (nostr-transport crate)

Define these as constants. All are ephemeral (kind >= 20000) except the
final published token.
KIND_SESSION_ANNOUNCE = 20001 // coordinator → signers: new signing session
KIND_ROUND1_COMMITMENT = 20002 // signer → coordinator: SigningCommitments
KIND_ROUND2_PAYLOAD = 20003 // coordinator → each signer: SigningPackage
KIND_PARTIAL_SIG = 20004 // signer → coordinator: SignatureShare
KIND_TIMESTAMP_TOKEN = 1 // coordinator → public: final signed token

### Event structure conventions

Every FROST event carries:

- tag `["session", "<session_id>"]` — 16-byte hex random ID per signing session
- tag `["p", "<recipient_npub>"]` — for routing and NIP-44 encryption

The `content` field of kinds 20001–20004 is always NIP-44 encrypted:

- 20001: coordinator encrypts to each signer's npub (send one event per signer)
- 20002: signer encrypts to coordinator npub
- 20003: coordinator encrypts to each signer's npub (one event per signer)
- 20004: signer encrypts to coordinator npub

The `content` of kind 1 (final token) is plaintext JSON.

### Relay subscription filters

Coordinator subscribes:

```json
{
  "kinds": [20002, 20004],
  "tags": { "p": ["<coordinator_npub>"] },
  "#session": ["<session_id>"]
}
```

Each signer subscribes on startup:

```json
{ "kinds": [20001, 20003], "tags": { "p": ["<my_npub>"] } }
```

## Signing message format

The coordinator constructs the message that gets signed. It must be
deterministic and commit to all fields that matter:
message = SHA-256("FROST-TIMESTAMP-V1\x00"
|| "serial=" || serial_number
|| "\x00hash=" || document_hash // "sha256:<hex>"
|| "\x00time=" || rfc3339_utc
|| "\x00policy=" || policy_string) // e.g. "2-of-3"

Return the raw 32-byte hash as the message passed to FROST (it expects a
`[u8; 32]` message).

## TimestampToken (common crate)

```rust
#[derive(Serialize, Deserialize)]
pub struct TimestampToken {
    pub version: u8,                      // always 1
    pub serial_number: u64,               // monotonic, coordinator-managed
    pub document_hash: String,            // "sha256:<hex>"
    pub timestamp: DateTime<Utc>,
    pub policy: String,                   // "2-of-3"
    pub k: usize,
    pub n: usize,
    pub signers: Vec<u16>,                // FROST identifier of each participating signer
    pub group_public_key: String,         // hex-encoded compressed point
    pub signature_r: String,             // hex-encoded R point
    pub signature_s: String,             // hex-encoded s scalar
}
```

Implement `verify(&self, document_bytes: &[u8]) -> bool` which:

1. Recomputes `document_hash` and checks it matches
2. Reconstructs the signing message
3. Calls `frost_secp256k1::verify_group_signature`

## Coordinator binary

### HTTP API (axum)

GET /api/v1/pubkey → { group_public_key, k, n, coordinator_npub }
POST /api/v1/timestamp → body: { "document": "<base64>" }
← TimestampToken JSON
POST /api/v1/verify → body: { "document": "<base64>", "token": {...} }
← { valid: bool, error?: string }
GET /health → { status: "ok" }

### Signing session orchestration

When `POST /api/v1/timestamp` arrives:

1. Compute `document_hash` and build the `TimestampToken` skeleton (no sig yet)
2. Derive the 32-byte signing message
3. Generate a random `session_id`
4. Select k signers from the authorised list (random selection; in future: health-aware)
5. For each selected signer, publish a NIP-44-encrypted `kind:20001` event to the relay pool
6. Subscribe to `kind:20002` filtered by `session_id` tag
7. Wait for k `SigningCommitments` to arrive (timeout: 30 seconds)
8. Build `SigningPackage` from all commitments + message
9. For each selected signer, publish a NIP-44-encrypted `kind:20003` with the `SigningPackage`
10. Subscribe to `kind:20004` filtered by `session_id`
11. Wait for k `SignatureShare`s to arrive (timeout: 30 seconds)
12. Call `frost_secp256k1::aggregate` → `Signature`
13. Verify the aggregated signature against the group public key — panic if invalid
14. Complete the `TimestampToken` with signature fields and serial number
15. Publish the token as `kind:1` to all relays
16. Return the token to the HTTP client

Handle timeouts by returning HTTP 503 with a message indicating which signers
did not respond. Do not retry automatically on the first version.

Use `tokio::sync::watch` or a `DashMap<session_id, SessionState>` to track
in-flight sessions. Only one session per document hash at a time.

## Signer-node binary

On startup:

1. Load config (KeyPackage, own nsec, coordinator npub, relay URLs)
2. Connect to relay pool
3. Subscribe to `kind:20001` and `kind:20003` filtered on own npub

Event loop:
on kind:20001 (session announcement):

- Decrypt content with own nsec
- Parse session_id, k, n, message
- Optionally show approval prompt if running interactively (--interactive flag)
- Call frost::round1::commit() → (SigningNonces, SigningCommitments)
- Store nonces in HashMap<session_id, SigningNonces> with creation timestamp
- Publish kind:20002 (commitments) encrypted to coordinator npub
  on kind:20003 (round2 payload):
- Decrypt content
- Look up stored nonces for session_id — if missing or expired, log and drop
- Call frost::round2::sign() → SignatureShare
- Delete nonces from map immediately (single use)
- Publish kind:20004 (partial sig) encrypted to coordinator npub

Nonce TTL: 5 minutes. Run a background task every 60 seconds to evict expired
sessions from the nonce map.

## Client CLI binary

frost-timestamp keygen --k 2 --n 3 --out ./configs
Generates key shares and writes node/coordinator config files.
Prints group public key to stdout.
WARN: trusted dealer — replace with DKG for production.
frost-timestamp timestamp <file> [--coordinator http://localhost:8000]
POSTs the file, writes <file>.tst token, prints summary.
frost-timestamp verify <file> <token.tst> [--coordinator http://localhost:8000]
Verifies token against file via coordinator API.
frost-timestamp inspect <token.tst>
Decodes and verifies token locally (no network). Prints all fields.
frost-timestamp pubkey [--coordinator http://localhost:8000]
Prints the group public key in hex and bech32.

## Error handling rules

- All errors in the signer event loop must be logged and recovered — the
  signer must never crash on a bad event from the relay
- Coordinator returns HTTP 400 for bad input, 503 for signing timeout,
  500 for internal errors with a JSON body `{ "error": "..." }`
- Nonce reuse must be impossible: always delete nonces from the map
  before returning from round2, regardless of success or failure
- Never log the KeyPackage secret or any scalar that could expose sk_i

---

## Out of scope for v1

- Proactive secret sharing / key refresh
- DKG (replace trusted dealer keygen)
- Signer authentication beyond Nostr keypair
- Relay authentication / NIP-42
- Persistent token storage (tokens are in Nostr relays and returned to client)
- Rate limiting on the coordinator HTTP API

## Acceptance criteria

1. `keygen --k 2 --n 3` produces valid configs; printed public key is a valid
   compressed secp256k1 point
2. With all 3 signers running, `timestamp <file>` completes in under 10 seconds
   and produces a `.tst` file
3. `inspect <file>.tst` verifies the signature locally without network access
4. Killing one signer (leaving 2 of 3 alive) does not break signing
5. Tampered document (change one byte) fails verification with a clear error
6. Replayed kind:20004 (send same partial sig twice for same session) is
   silently ignored — only the first is used
7. Session timeout (all signers killed mid-session) returns HTTP 503 within
   35 seconds, no panic, signer nodes recover and handle the next session
